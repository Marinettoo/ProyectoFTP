1. Objetivo

El objetivo de este proyecto es que los estudiantes trabajen en equipo para configurar un servidor FTP utilizando herramientas de virtualización y automatización como Vagrant y Ansible. El trabajo será gestionado mediante un repositorio en GitHub, donde cada integrante del equipo asumirá responsabilidades específicas y contribuirá al éxito del proyecto.

2. Descripción de la tarea

  • Creación del repositorio en Github
  • Configuración del proyecto en Github

3. Creación del Repositorio en GitHub

Configurar un repositorio en GitHub para centralizar el trabajo del equipo.

3.1. Incluir una licencia adecuada para el proyecto.

Un archivo README.md con información básica del proyecto:

  • Descripción del objetivo.
  • Herramientas utilizadas.
  • Instrucciones iniciales de uso.
  • Roles de los integrantes del equipo.

4. Configurar las herramientas de gestión de proyectos de GitHub

Crear un tablero de proyectos (GitHub Projects) para planificar y hacer seguimiento de las tareas.

  • Definir columnas como Por hacer, En progreso y Hecho.
  • Asignar las tareas de manera equitativa entre los integrantes y usar etiquetas para organizar el trabajo.

5. Creación de la Infraestructura con Vagrant

Configurar un entorno virtualizado mediante Vagrant que incluya:

  • Un archivo Vagrantfile para definir la máquina virtual.
  • Configuración de red y recursos de la máquina virtual según las necesidades del servidor FTP.
  • Documentar el proceso en el repositorio.

6. Creación de la Infraestructura de Ansible

Desarrollar los playbooks y roles de Ansible necesarios para configurar automáticamente el servidor FTP.

Configurar:

  • Instalación del software del servidor FTP.
  • Configuración de usuarios y permisos.
  • Políticas de seguridad y opciones avanzadas.
  • Asegurar que los playbooks sean reutilizables y bien documentados.

7. Pruebas

7.1. ansible-lint

Verificar la calidad y estilo de los playbooks.

7.2. Realizar pruebas funcionales del servidor FTP

  • Conexión al servidor desde un cliente FTP.
  • Transferencia de archivos.
  • Validación de usuarios y permisos configurados.
  • Documentar los resultados de las pruebas en el repositorio.

8. Documentación

Redactar una guía detallada que incluya:

  • Pasos para desplegar el servidor desde el repositorio.
  • Ejemplos de uso del servidor FTP.
  • Resolución de problemas comunes.
  • Consideraciones de diseño y justificación técnica.

10. Criterios de Evaluación

Funcionamiento del Servidor FTP
El servidor debe estar correctamente configurado y cumplir con los requisitos funcionales.
Calidad del Código
Uso adecuado de Vagrant, Ansible y Git, además de la correcta implementación de buenas prácticas.
Gestión del Proyecto
Uso eficiente de las herramientas de GitHub para planificar, organizar y documentar el progreso del equipo.
Organización y Colaboración
Uso eficiente del tablero de GitHub Projects, con tareas bien definidas y actualizadas regularmente.
Documentación
La guía debe ser clara, completa y profesional.
Pruebas
Resultados de las pruebas funcionales y de estilo.

11. Plazo de Entrega

  • El proyecto deberá entregarse antes de la fecha que aparece en la tarea de Moodle, incluyendo el repositorio final en GitHub y un informe/documentación en formato PDF.
  • Después de esta fecha se podrá seguir entregando durante una semana aunque reduciendo la nota.

12. Rúbrica de Evaluación

12.1. Funcionamiento del Servidor FTP

  • El servidor FTP funciona correctamente y permite conexiones externas. 2 puntos.
  • Se ha configurado correctamente el acceso, usuarios, y permisos según lo especificado. 2 puntos.
  • El servidor permite la transferencia de archivos y cumple los requisitos funcionales mínimos. 1 punto.

12.2. Calidad del Código

  • Los archivos de configuración (Vagrantfile, playbooks de Ansible) están correctamente estructurados y sin errores sintácticos. 2 puntos.
  • Los playbooks y roles de Ansible siguen buenas prácticas, verificado con ansible-lint. 1 punto.
  • El código es modular, reutilizable y bien comentado. 1 punto.

12.3. Gestión del Proyecto

  • El repositorio incluye una estructura clara con un README.md bien elaborado (objetivos, instrucciones, roles del equipo, etc.). 2 puntos.
  • Uso de GitHub Projects: tablero con tareas definidas y actualizadas periódicamente (estado: Por hacer, En progreso, Hecho). 2 puntos.
  • Buen uso del control de versiones (commits frecuentes, mensajes descriptivos, ramas, pull requests bien gestionados). 2 puntos.

12.4. Pruebas

  • Pruebas funcionales realizadas (verificación de conexión, transferencia de archivos, usuarios y permisos) y documentadas. 2 puntos.
  • Ejecución de `ansible-lint` sin errores graves y evidencia en el repositorio. 1 punto.

12.5. Documentación

  • Guía clara y completa para desplegar el servidor FTP y utilizarlo. 2 puntos.
  • Ejemplos prácticos incluidos en la documentación (por ejemplo, conexión al servidor, comandos FTP). 1 punto.

12.6. Trabajo en Equipo

  • Evidencia de colaboración equitativa entre los miembros del equipo (reparto de tareas reflejado en el tablero de GitHub Projects y el historial de commits). 2 puntos.
  • Comunicación efectiva y resolución conjunta de problemas. 1 punto.

13. Plazo de Entrega

El proyecto deberá entregarse antes de la fecha de entrega que aparece en la tarea de Moodle, incluyendo el repositorio final en GitHub y un informe/documentación en formato PDF.

14. Ejemplo de Organización del Tablero de GitHub Projects

A continuación, se muestra cómo estructurar el tablero de GitHub Projects para gestionar su proyecto. El objetivo es visualizar el estado de cada tarea, asignar responsables y mantener el progreso organizado.

14.1. Estructura del Tablero de Proyectos

  • Columnas:
    • To Do (Por Hacer): Tareas pendientes de iniciar.
    • In Progress (En Progreso): Tareas en las que se está trabajando activamente.
    • Review (Revisión): Tareas completadas que necesitan revisión o pruebas.
    • Done (Hecho): Tareas finalizadas y revisadas.
  • Etiquetas (usa etiquetas de colores para organizar las tareas):
    • Infraestructura: Tareas relacionadas con Vagrant y configuración del entorno.
    • Ansible: Creación de playbooks y roles.
    • Pruebas: Pruebas funcionales y de calidad.
    • Documentación: Redacción del README y guía de usuario.
    • General: Gestión del repositorio, organización del equipo, etc.
  • Asignación de Tareas:
    • Cada tarea debe asignarse a un responsable (o varios, si es necesario).
    • Incluye una breve descripción y enlaces relacionados, si aplica.

16. Tarea Adicional: Configuración del Servidor DNS

La tarea de configuración del servicio DNS se puede realizar de dos formas. La más sencilla sería falsear la consulta utilizando direcciones del servicio nip.io. Por ejemplo ftp.192.168.57.10.nip.io con lo cual no haría falta el servicio DNS. La otra manera, la real, se realizaría a continuación y sería una tarea adicional.

16.1. Configuración del DNS con Ansible

  • Crear un playbook de Ansible para configurar un servidor DNS (por ejemplo, utilizando BIND9).
  • Añadir un registro A en la zona DNS para resolver el dominio del servidor FTP. Ejemplo: ftp.example.test -> 192.168.56.10 (IP de la máquina virtual de Vagrant).
  • Configurar un archivo de zona con los siguientes registros básicos:
    • Registro A: Dirección IP del servidor FTP.
    • Registro CNAME (opcional): Alias para simplificar referencias al dominio.
    • Registro PTR: Resolución inversa de IP a nombre de dominio (opcional).

16.2. Integración del DNS en el Proyecto

Modificar la configuración de red del cliente para que utilice el servidor DNS configurado.

  • Documentar los pasos necesarios para probar la resolución del dominio:
    • Uso del comando nslookup o dig para validar que ftp.example.test se resuelve correctamente.
    • Pruebas de conexión al servidor FTP utilizando el dominio configurado en lugar de la IP.

17. Evaluación

Criterios Puntos
El servidor DNS se configura correctamente y resuelve el dominio `ftp.example.com`. 2
Los registros A, CNAME (opcional) y PTR (opcional) están configurados correctamente. 1
La configuración está integrada con el servidor FTP y permite conexiones usando el dominio. 1
